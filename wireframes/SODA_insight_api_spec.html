1콘텐츠 생성 트리거 / 갱신 주기
항목내용
생성 방식서버에서 정기 배치(예: 매일 1회) 또는 사용자 행동 트리거(예: 신규 가입 시 초기 추천) 기준으로 LLM API 호출하여 콘텐츠 생성·저장
"오늘의 추천" 개인화사용자의 5축 행동성격 데이터 + 최근 활동(참여 챌린지 카테고리 등)을 프롬프트 컨텍스트로 활용해 맞춤 콘텐츠 우선순위 결정
저장 방식생성된 콘텐츠는 DB에 저장 후 캐싱 — 매 화면 진입마다 LLM을 재호출하지 않음 (비용/속도 이슈 방지)
중요: 클라이언트(앱)는 LLM API를 직접 호출하지 않음. 서버가 생성·검수 후 일반 REST 응답으로 클라이언트에 내려주는 구조를 권장 (실시간 생성 시 응답 지연 및 품질 검수 불가 문제 발생)
2상단 영역 — 카테고리 / 제목
필드설명
categoryAI가 생성 시 사전 정의된 카테고리 목록(크래프트/커리어/루틴/직무탐색/트렌드) 중 하나로 분류하도록 프롬프트에 강제. 카테고리별 색상은 클라이언트에서 프리셋 매핑
title20자 내외 권장. 너무 길 경우 2줄까지만 노출하고 말줄임(...) 처리

필드설명
createdAt콘텐츠 생성일, YYYY.MM.DD 포맷으로 표시
readingTimeMinAI 응답이 아닌 클라이언트/서버에서 본문 글자수 기준 자동 계산(예: 분당 400자 기준). API가 직접 줄 필요 없음
viewCount실제 사용자 조회 시 서버에서 카운트 증가, AI 생성값 아님

항목내용
생성 방식별도 이미지 생성 API(예: DALL-E 등) 연동 또는 카테고리별 고정 일러스트/그라데이션 프리셋 사용 — 비용 효율 위해 후자 권장
대체(fallback)이미지 생성 실패 시 카테고리 컬러 기반 그라데이션 블록으로 자동 대체

필드설명
bodyBlocks본문은 단일 긴 텍스트가 아니라 블록 배열(문단/소제목 타입 구분)로 받아서, 클라이언트가 소제목/문단을 구분해 스타일링
길이 가이드프롬프트에 "본문 3~5개 문단, 소제목 1~2개 포함" 등으로 구조 강제 권장 (일관된 레이아웃 유지를 위함)
강조 텍스트중요 키워드는 마크다운 굵게 표시 형태로 받아 클라이언트에서 강조 스타일로 변환
AI 생성 콘텐츠는 반드시 발행 전 검수 단계(자동 필터링 + 운영자 검토)를 거치도록 권장 — 잘못된 정보나 부적절한 표현이 사용자에게 그대로 노출되는 리스크 방지
